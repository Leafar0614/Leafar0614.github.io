Android系统架构

Android系统架构

首先，系统架构分为五层。从上到下依次是应用层、应用框架层、系统运行库层、硬件抽象层 和 Linux内核层。

应用层

内置和非内置应用程序都属于应用层。

应用框架层（Java Framework)

应用框架层为开发者提供了所需的API，平时开发就是调用这一层提供的API，由JAVA编写，一下是主要的组件：

名称 功能描述
Activity Manager(活动管理器) 管理各个应用程序生命周期
Location Manager(位置管理器) 提供定位功能服务
Package Manager(包管理器) 管理所有安装在系统中的应用程序
Notification Manager(通知管理器) 使得应用程序可以在状态栏中显示自定义的提示信息
Resource Manager（资源管理器） 管理app使用的各种非代码资源(本地化字符串、图片、布局文件、颜色文件等)
Telephony Manager(电话管理器) 管理电话功能
Window Manager（窗口管理器） 管理所有开启的窗口程序
Content Providers（内容提供器） 不同应用程序之间数据共享
View System（视图系统） 构建应用程序的基本组件

系统运行库层（Native)

这里分为两部分 分别为C/C++程序库和Android运行时库。

1.C/C++程序库

C/C++ 程序库能被Android系统中的不同组件使用，并通过应用程序框架为开发者提供服务，主要的C/C++程序库有：

名称 功能描述
OpenGL 3D绘图函数库
Libc 从BSD继承来的标准C系统函数库，专门为基于嵌入式Linux的设备定制（不是太懂）
Media Framework 多媒体库，支持多种常用的音频、视频格式录制和回放。
SQLite 轻型的关系型数据库库
SGL 底层的2D图形渲染引擎
SSL 安全套接层，是为网络通信提供安全及数据完整性的一种安全协议
FreeType 字体引擎

2.Android运行时库

运行时库又分为核心库和ART(5.0之后，Dalvik虚拟机被ART取代)。
核心库提供了JAVA语言大多数功能。
相较于JVM虚拟机，Dalvik是专门为移动设备定制的，允许在有限的内存中同时运行多个虚拟机的实例。并且每一个Dalvik应用作为一个独立的Linux进程执行。（独立的进程可以防止在虚拟机奔溃的时候其他程序被关闭）
ART的机制与Dalvik有所不同。在Dalvik下，应用每次运行的时候，字节码都需要通过即时编译器转换为机器码，这会拖慢应用的运行效率，而在ART环境中，应用在第一次安装的时候，字节码就会预先编译成机器码，成为本地应用。

硬件抽象层（HAL）

硬件抽象层是位于操作系统内核与硬件电路之间的接口层，目的在于将目的抽象化，为了保护硬件厂商知识产权，它隐藏了特定平台的硬件接口细节，将控制硬件的动作放在硬件抽象层中。为操作系统提供虚拟硬件平台，使其具有硬件无关性，可在多种平台上进行移植。

Linux内核层

Android 的核心系统服务基于Linux 内核，在此基础上添加了部分Android专用的驱动。系统的安全性、内存管理、进程管理、网络协议栈和驱动模型等都依赖于该内核。
